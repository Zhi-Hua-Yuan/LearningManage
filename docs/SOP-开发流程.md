# SOP-开发流程（LearningManage）

## 1. 目的
- 统一模块开发节奏，确保需求可追踪、实现可验证、结果可交接。
- 约束范围膨胀，降低返工成本。

## 2. 适用范围
- 本项目所有模块开发（后端优先）：Project / Milestone / Task / WeeklyReview / Dashboard / AI占位。

## 3. 标准流程

### 步骤 1：需求定义
- 使用模板：`docs/codex-templates/01-需求说明模板.md`
- 落地路径：`docs/modules/<模块名>/requirement.md`
- 必填项：
  - 背景、目标、非目标
  - 功能需求
  - 接口与数据库变更
  - 验收标准（Given/When/Then）
- 进入下一步条件：
  - 非目标明确
  - 验收标准可测试

### 步骤 2：TODO拆解
- 使用模板：`docs/codex-templates/02-TODO清单模板.md`
- 落地路径：`docs/modules/<模块名>/todo.md`
- 必含任务类型：
  - 后端接口
  - 数据库
  - 参数校验/异常处理
  - 测试
  - 文档更新
- 要求：
  - 明确优先级（P0/P1）
  - 明确状态（待开始/进行中/阻塞/已完成）
  - 明确执行顺序

### 步骤 3：Codex执行
- 使用模板：`docs/codex-templates/03-Codex执行任务模板.md`
- 输入要求：
  - 模块名、仓库路径、分支、需求文档、TODO文档
  - 本次目标（只写本轮范围）
  - 明确不做（至少 3 条）
  - 技术约束（版本、返回结构、异常约束、数据库约束）
- Codex执行要求：
  - 按 TODO 顺序逐项完成
  - 每项说明改动文件和原因
  - 完成后执行校验（至少编译）
  - 输出变更摘要、风险点、后续建议

### 步骤 4：模块审阅
- 使用模板：`docs/codex-templates/04-模块审阅模板.md`
- 落地路径：`docs/modules/<模块名>/review.md`
- 审阅重点：
  - 需求达成与验收一致性
  - 空指针/逻辑漏洞/并发与事务风险
  - 参数校验、异常处理、错误码规范
  - SQL 与索引合理性
  - 测试覆盖缺口
- 输出要求：
  - 问题清单按高/中/低分级
  - 测试缺口
  - 非阻塞优化建议

### 步骤 5：修复与交接
- 使用模板：`docs/codex-templates/05-模块交接文档模板.md`
- 落地路径：`docs/modules/<模块名>/handover.md`
- 必填项：
  - 模块概览、核心类职责
  - 接口清单（方法/入参/出参/错误码）
  - 数据库变更说明
  - 测试结果与已知限制
  - 风险与监控建议

### 步骤 6：PR提交
- 使用：`.github/PULL_REQUEST_TEMPLATE.md`
- PR最小内容：
  - 需求文档链接
  - TODO文档链接
  - 主要改动
  - 验证结果
  - 风险与回滚说明

## 4. 目录规范
- `docs/modules/<模块名>/requirement.md`
- `docs/modules/<模块名>/todo.md`
- `docs/modules/<模块名>/review.md`
- `docs/modules/<模块名>/handover.md`

## 5. DoD（完成定义）
- [ ] 代码实现完成且编译通过
- [ ] 核心路径测试通过（自动或手工）
- [ ] 边界场景已验证
- [ ] 返回结构与异常码符合项目规范
- [ ] 需求/TODO/审阅/交接文档齐全

## 6. 协作约定（你与Codex）
- 你每次输入必须包含：
  - 模块名
  - 当前文档路径
  - 本次目标
  - 明确不做
- Codex每次输出必须包含：
  - 变更文件列表
  - 核心实现说明
  - 验证结果
  - 风险与边界
  - 下一步建议

## 7. 当前项目执行顺序建议
1. 先完成 Project 模块闭环
2. 再做 Milestone 与 Task
3. 然后补 WeeklyReview 与 Dashboard 数据接口
4. 最后接入 AI 能力（先占位后真实集成）
